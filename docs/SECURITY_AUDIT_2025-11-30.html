Executive Summary

This security audit of ZipherX iOS wallet examined five critical areas: key storage, network layer, transaction building, database persistence, and input validation. The audit identified 53 security vulnerabilities across these domains, with 12 CRITICAL issues requiring immediate attention.

Key Findings

  • Key Storage: Spending keys stored unencrypted in SQLite database
  • Network: Equihash PoW verification disabled, no certificate pinning
  • Transactions: Spending key not kept in Secure Enclave during signing
  • Database: PIN stored in plaintext, no SQLCipher encryption
  • Input: Amount field vulnerable to integer overflow

Critical Vulnerabilities

Spending Key Stored Unencrypted in Database CRITICAL
WalletDatabase.swift:90,319-336 | Category: Key Storage
The spending key (96 bytes containing ask+nsk+ovk) is stored directly in the SQLite accounts table as a BLOB without any encryption layer. Although iOS Data Protection encrypts the file at rest, an attacker with access to an unlocked device can read the plaintext key.
CREATE TABLE accounts (
  spending_key BLOB NOT NULL, -- UNENCRYPTED!
  viewing_key BLOB NOT NULL
);

// RECOMMENDATION

Remove spending_key and viewing_key columns. Derive keys from Secure Enclave on each use. Never persist cryptographic secrets in database.

Equihash Proof-of-Work Verification Disabled CRITICAL
HeaderSyncManager.swift:327-331 | Category: Network Security
Block headers are accepted without cryptographic proof-of-work verification. The Equihash(200,9) verification is deliberately disabled with a TODO comment. This allows a malicious peer to feed arbitrary fake headers to the wallet.
let header = try ZclassicBlockHeader.parseWithSolution(
  data: fullHeaderData,
  height: height,
  verifyEquihash: false // DISABLED!
)

// RECOMMENDATION

Fix Equihash verification implementation. Add proper error handling instead of silently disabling. Temporarily sync headers from trusted checkpoints until verification is fixed.

No Certificate Pinning for InsightAPI CRITICAL
InsightAPI.swift:8 | Category: TLS/SSL
All HTTPS requests to explorer.zcl.zelcore.io use default certificate validation with no pinning. This enables man-in-the-middle attacks if a CA is compromised or through network-level interception.

// RECOMMENDATION

Implement certificate pinning using public key SHA256 hash. Add backup pins for certificate rotation.

Insufficient Header Consensus Threshold CRITICAL
HeaderSyncManager.swift:15-16 | Category: Network Consensus
Headers are accepted from a single peer (minPeers = 1, consensusThreshold = 1). A single malicious peer can completely control the blockchain view, enabling trivial eclipse attacks.

// RECOMMENDATION

Change minPeers to at least 3-6. Require majority agreement (N/2 + 1) for header consensus.

Spending Key Exposed in Memory During Signing CRITICAL
TransactionBuilder.swift:73,281-292 | Category: Key Management
The spending key is retrieved from Secure Enclave storage, then held in unprotected Swift Data memory during transaction building. The key is passed through multiple function layers and never zeroed after use.

// RECOMMENDATION

Keep spending key in Secure Enclave throughout transaction signing. Use SecKeyCreateDecryptedData() to perform signing inside the enclave. Implement SecureData wrapper that zeros memory on deinit.

PIN Stored in Plaintext in UserDefaults CRITICAL
SettingsView.swift:1297 | Category: Authentication
The wallet PIN code is stored in plaintext in UserDefaults. UserDefaults is not encrypted and is accessible to any process with device access. The code even contains a comment: "in production, hash this!"
// Line 1296: "in production, hash this!"
UserDefaults.standard.set(pinCode, forKey: "walletPIN")

// RECOMMENDATION

Use PBKDF2 or bcrypt to hash PIN before storing. Store hashed PIN in Keychain, not UserDefaults. Add rate limiting on PIN verification attempts.

Sensitive Data (Nullifiers) Logged to Console CRITICAL
FilterScanner.swift, TransactionBuilder.swift | Category: Information Disclosure
Nullifiers (cryptographic commitments that uniquely identify spent notes) are logged in plaintext. This compromises transaction privacy and allows linking transactions to specific users.

// RECOMMENDATION

Remove ALL logging of nullifiers, spending keys, and other cryptographic values. Use compile-time DEBUG_LOGGING flag that is FALSE in production builds.

Database Not Encrypted with SQLCipher CRITICAL
WalletDatabase.swift:60-62 | Category: Encryption
The SQLite database relies only on iOS Data Protection, which only encrypts when the device is locked. The code comment mentions SQLCipher "can be added" but it's not implemented. Database contains spending keys, nullifiers, transaction history, and balance information.

// RECOMMENDATION

Implement SQLCipher encryption layer with key stored in Secure Enclave. Use PRAGMA key='...' for database encryption. All database contents will then be encrypted at rest.

Amount Field Integer Overflow CRITICAL
SendView.swift:455 | Category: Input Validation
Conversion from Double to UInt64 for amount field can cause silent overflow. If amountValue * 100,000,000 exceeds UInt64.max, the conversion silently wraps/truncates, potentially causing incorrect transactions.
let zatoshis = UInt64(amountValue * 100_000_000) // OVERFLOW RISK!

// RECOMMENDATION

Add explicit bounds checking before conversion. Use Decimal instead of Double for financial amounts. Validate maximum allowed amount before transaction building.

Anchor/Witness Can Be All-Zeros CRITICAL
TransactionBuilder.swift:197,252 | Category: Proof Validation
Both anchor and witness can silently become all-zeros without detection. A placeholder with zero-filled anchor is created if header lookup fails, and the check happens AFTER witness rebuild.

// RECOMMENDATION

Use Optional<Data> and require explicit anchor before transaction building. Add explicit validation that anchor is not all-zeros before passing to FFI layer.

Private Key Export Without Fresh Biometric Auth CRITICAL
WalletManager.swift:1673 | Category: Authentication
The spending key can be exported without requiring fresh biometric authentication. The exported key is stored in a @State variable and copied to clipboard, where it persists and can be read by other apps.

// RECOMMENDATION

Require fresh Face ID for every key export. Never store exported key in @State. Clear clipboard automatically after 30 seconds. Warn user about clipboard persistence.

Weak Database Encryption Key Derivation CRITICAL
WalletManager.swift:145,518 | Category: Key Derivation
The database encryption key is derived using SHA-256 directly from the spending key without salt. SHA-256 is not a key derivation function (KDF), making rainbow table attacks feasible if spending key is leaked.
let dbKey = Data(SHA256.hash(data: spendingKey)) // No salt, no KDF!

// RECOMMENDATION

Use proper KDF like PBKDF2 with random salt and 100,000+ iterations. Or use Argon2id for memory-hard key derivation. Store salt in Keychain.

Conclusion

ZipherX demonstrates a solid architectural foundation for a privacy-focused mobile wallet, with proper use of Sapling cryptography, shielded-only transactions, and decentralized P2P networking. However, the current implementation contains 12 critical security vulnerabilities that must be addressed before any production deployment.

The most urgent issues relate to key storage (unencrypted keys in database), network security (disabled PoW verification), and authentication (plaintext PIN). These vulnerabilities could enable complete wallet compromise and fund theft.
