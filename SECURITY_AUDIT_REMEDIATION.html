TASK 03 Disable DEBUG_LOGGING in Rust FFI
CRITICAL OPEN ▶

Vulnerability

VUL-F-004

Severity Upgrade

Upgraded from HIGH to CRITICAL — IVK, shared secrets, and KDF material are actively written to an unencrypted log file readable by any local process. This is equivalent to a live key exfiltration channel.

Evidence

Line 13 of lib.rs: const DEBUG_LOGGING: bool = true; — still enabled. The debug_log! macro guards ~220+ log calls but they ALL fire. Additionally, ~324 unguarded eprintln!() calls print regardless.

Sensitive Data Actively Logged

  • Full IVK scalar (lines 863, 921)
  • Full EPK (line 920)
  • pk_d bytes (line 871)
  • Shared secret material (line 917)
  • KDF key material (line 935)
  • Diversifier plaintext (line 947)
  • RCM values (line 6335)
  • Full TX hex (line 6523)
  • Nullifier values (line 2358)

Libraries/zipherx-ffi/src/lib.rs : line 13

Required Changes

  1. Set const DEBUG_LOGGING: bool = false;
  2. Wrap ALL eprintln!() calls containing sensitive data in if DEBUG_LOGGING {}
  3. Use compile-time gating for production:
#[cfg(debug_assertions)]
const DEBUG_LOGGING: bool = true;
#[cfg(not(debug_assertions))]
const DEBUG_LOGGING: bool = false;
Requires recompiling the Rust FFI library.

TASK 05 Fix Buffer Overflow in Parallel Decryption
CRITICAL PARTIAL ▶

Vulnerability

VUL-F-002

What's Fixed

Input validation via safe_slice() at lines 1070-1108. Pre-parsing uses safe slice indexing. Integer overflow protection using saturating_sub pattern confirmed in 6 tree-loading functions.

What Remains

  1. Lines 1095 and 1102 use output_count * 644 and output_count * 564 with NO checked_mul() — zero checked_mul calls exist in the entire file
  2. Lines 1150-1174 use raw pointer arithmetic (out_ptr.add(result_offset), ptr::copy_nonoverlapping) bypassing Rust's borrow checker

Libraries/zipherx-ffi/src/lib.rs : lines 1058-1180

Required Changes

let total_output_size = output_count.checked_mul(644).unwrap_or(0);
if total_output_size == 0 && output_count > 0 { return 0; }
let total_result_size = output_count.checked_mul(564).unwrap_or(0);
if total_result_size == 0 && output_count > 0 { return 0; }

TASK 25 Fix macOS Key Access Security
CRITICAL OPEN ▶

Vulnerability

VUL-U-007

Severity Upgrade

Upgraded from MEDIUM to CRITICAL — Hardware UUID is publicly readable via ioreg by any local process. Combined with the on-disk .enc file and salt, this is a deterministic spending key extraction path requiring zero privileges beyond same-user access. Any macOS malware running as the user can silently steal all funds.

Evidence

ALL macOS builds bypass Secure Enclave. File-based storage uses AES-GCM with encryption key derived from Hardware UUID via IOKit kIOPlatformUUIDKey — publicly readable by any process. Key retrieval requires ZERO authentication.

Any process running as the same user can read the .enc file + salt + Hardware UUID → derive key via HKDF-SHA256 → decrypt spending key.

Required Changes

  1. Migrate to macOS Keychain Services with kSecAttrAccessibleWhenUnlockedThisDeviceOnly
  2. Add kSecUseUserPresence context for biometric/password gating
  3. Create migration: read existing .enc key, store in Keychain, securely delete .enc file
  4. Fallback: if Keychain unavailable, use password-derived key (PBKDF2/Argon2) instead of Hardware UUID
MUST be completed BEFORE TASK 2 (removing spending key from DB) — the DB copy is the only remaining backup if macOS key storage fails.

Sources/Core/Storage/SecureKeyStorage.swift : lines 73-80, 335-476, 398-437
